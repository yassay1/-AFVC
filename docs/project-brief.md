# AFC RiskOps Agent System — 项目说明书

> 项目定位、目标用户、核心价值与面试讲解口径。

---

## 1. 项目一句话介绍

面向地铁 AFC（自动售检票）设备的智能运维系统：基于历史故障工单数据预测复发风险，通过 Agent 自动整合设备历史、风险预测、预警等级和维修建议，生成可解释的运维诊断报告。

---

## 2. 解决的问题

| 痛点 | 本系统方案 |
|------|-----------|
| 历史工单多，难以判断哪些设备易复发 | 多时间窗口风险预测 + 红橙黄绿预警 + Top N 排序 |
| 运维人员不知道今天优先巡检什么 | 高风险设备列表 + 巡检窗口建议 |
| 单设备异常后需快速了解全貌 | 综合分析聚合历史、风险、预警、建议 |
| 预测值不直观 | Agent 将风险值翻译为业务可理解的预警和建议 |
| 维修建议依赖经验 | 基于历史故障描述的关键词规则 + 未来 RAG 扩展 |

---

## 3. 目标用户与使用场景

### 运维主管 — "今天优先巡检哪些设备？"
- 高风险设备 Top N 列表
- 每台设备的车站、线路、品牌、6 时间窗口风险
- 红橙黄绿预警 + 建议巡检窗口

### 一线维修人员 — "帮我分析设备 XXX，风险高不高，应该检查什么？"
- 设备基础信息 + 历史工单摘要
- 30/90 天风险 + 预警等级
- 维修检查方向 + 备件建议 + 科学边界

### 设备管理人员 — "为什么这台设备是红色预警？"
- 风险值是否超过阈值
- 历史工单数量和近期密度依据
- 高频故障现象 + 巡检建议

---

## 4. 系统主流程

```
用户上传 AFC 工单 Excel/CSV → Polars 读取分析
    → 数据概览 + 设备列表 + 单设备历史
    → 多时间窗口复发风险预测 (Mock/外部模型)
    → 红橙黄绿预警 + 维修建议 + 综合分析
    → Agent 调用工具整合 → 可解释诊断报告
```

---

## 5. Agent 核心价值

**不用 Agent：** 用户需分别查看 6 个页面，自行整理结论。

**使用 Agent 后：** 自然语言提问 → Agent 自动完成设备识别 → 工具选择 → 风险预测 → 预警判断 → 历史摘要 → 维修建议 → 诊断报告。

> Agent 把多个页面、多个接口、多个工具的结果，自动组织成面向运维决策的诊断结论。

---

## 6. 为什么不设计多子 Agent？

1. AFC 运维诊断链路清晰，不需要多 Agent 自由协商
2. 多 Agent 增加状态传递、Prompt 管理和调试复杂度
3. 业务服务层已拆分清楚，不需要再包装成多个子 Agent
4. 工业场景更需要稳定、可追踪、可解释

**最终设计：** 一个 AFCDiagnosisAgent × 8 个 LangChain Tools × LangGraph 工作流 × LangSmith 可观测

## 8. 面试讲解口径

> 这个项目是一个面向地铁 AFC 设备的智能运维系统。它基于历史故障工单数据，预测设备在未来 7/14/21/30/60/90 天内再次产生故障工单的风险，根据 30 天和 90 天风险生成红橙黄绿预警。
>
> 我的重点不是训练预测模型，而是做系统工程和 Agent 编排。我把数据概览、设备历史、风险预测、预警等级、维修建议封装成 LangChain Tools，再用 LangGraph 设计一个 AFC 诊断 Agent。用户用自然语言询问某台设备风险高不高、为什么触发预警、应该先检查什么，Agent 调用对应工具，整合证据，生成结构化诊断报告。
>
> 同时接入 LangSmith 追踪每次 Agent 调用了哪些工具、拿到了什么结果、最终报告是否忠实使用工具输出，保证系统不是黑盒聊天，而是可追踪、可解释、可评估的运维诊断流程。

---

## 9. 科学边界声明

1. `current_faildate` 是工单记录时间，不直接等同于物理故障发生时刻
2. 风险预测表示未来时间窗口内再次产生故障工单的风险，不等于设备一定会在某天发生故障
3. 维修建议是巡检方向参考，不是最终根因诊断结论
4. 最终维修判断仍需结合现场检测、设备日志和人工经验

- 设备识别正确、工具调用正确
- 报告引用工具结果，不编造数据
