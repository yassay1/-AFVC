# AFC 故障复发风险预测与智能维修建议系统：项目解释文件

> 本文件用于统一项目定位、用户使用场景、核心目标、功能边界和面试讲解口径。  
> 项目主线：**工单数据 → 设备风险预测 → 红橙黄绿预警 → 单设备综合分析 → AFC 诊断 Agent 报告生成**。

---

## 1. 项目名称

**AFC 故障复发风险预测与智能维修建议系统**

英文可命名为：

**AFC RiskOps Agent System**

---

## 2. 项目一句话介绍

本项目是一套面向地铁 AFC（Automatic Fare Collection，自动售检票）设备的智能运维系统，基于历史故障工单数据，预测设备在未来多个时间窗口内再次产生故障工单的风险，并通过 Agent 自动整合设备历史、风险预测、预警等级和维修建议，生成可解释的运维诊断报告。

---

## 3. 项目真正解决的问题

地铁 AFC 设备运维中，常见问题不是“有没有工单”，而是：

1. 历史故障工单很多，人工难以及时判断哪些设备更容易复发；
2. 运维人员需要知道“今天应该优先巡检哪些设备”；
3. 单台设备出现异常后，需要快速了解它的历史故障、复发风险和建议检查方向；
4. 预测模型输出风险值后，普通用户不一定理解风险值的业务含义；
5. 维修建议不能只靠大模型自由发挥，必须基于历史工单、预测结果和规则/知识依据。

因此，本项目的价值不是单纯做数据展示，也不是简单聊天机器人，而是：

> 把历史工单、预测模型、预警规则和维修建议整合成一个能辅助运维决策的智能诊断系统。

---

## 4. 目标用户

### 4.1 运维主管

关注问题：

- 今天哪些 AFC 设备最需要巡检？
- 哪些设备是红色/橙色预警？
- 巡检优先级如何安排？

典型用法：

> “今天最需要优先巡检的设备有哪些？”

系统应返回：

- 高风险设备列表；
- 每台设备的车站、线路、品牌；
- 7/14/21/30/60/90 天复发风险；
- 红橙黄绿预警等级；
- 建议巡检窗口。

---

### 4.2 一线维修人员

关注问题：

- 某台设备以前出过什么故障？
- 当前故障是否可能复发？
- 现场应该先检查什么？

典型用法：

> “帮我分析一下设备 100023，未来一个月风险高不高，如果风险高应该先检查什么？”

系统应返回：

- 设备基础信息；
- 历史工单摘要；
- 30 天风险与 90 天风险；
- 预警等级；
- 建议检查方向；
- 维修建议；
- 科学边界说明。

---

### 4.3 设备管理人员

关注问题：

- 为什么某台设备被判为红色预警？
- 风险判断依据是什么？
- 如何向上级解释系统结论？

典型用法：

> “为什么设备 100023 是红色预警？”

系统应返回：

- 30 天风险是否超过阈值；
- 90 天风险是否超过阈值；
- 历史工单数量是否偏高；
- 近期工单密度是否偏高；
- 高频故障现象是什么；
- 巡检建议是什么。

---

## 5. 系统主流程

```text
用户上传 AFC 故障工单 Excel / CSV
        ↓
系统保存原始工单文件
        ↓
Polars 读取并分析工单数据
        ↓
生成数据概览、设备列表、单设备历史
        ↓
prediction_service 计算多时间窗口复发风险
        ↓
warning_service 生成红/橙/黄/绿预警等级
        ↓
advice_service 生成维修检查方向
        ↓
analysis_service 聚合单设备综合分析
        ↓
AFCDiagnosisAgent 调用工具并生成诊断报告
        ↓
Streamlit 前端展示结果
```

---

## 6. 核心功能范围

### 6.1 数据上传

支持上传：

- `.xlsx`
- `.xls`
- `.csv`

上传后保存到：

```text
backend/data/raw/
```

---

### 6.2 数据概览

展示：

- 工单总数；
- 设备数量；
- 车站数量；
- 线路数量；
- 品牌数量；
- 工单时间范围；
- 品牌分布；
- 线路分布；
- 高频故障描述 Top N；
- 工单类型分布。

---

### 6.3 设备管理

支持：

- 查看设备列表；
- 按工单数量排序；
- 查询单设备历史工单；
- 查看设备所属车站、线路、品牌、子系统；
- 查看最近工单记录时间。

---

### 6.4 风险预测

针对单台设备输出 6 个时间窗口风险：

```text
7 天 / 14 天 / 21 天 / 30 天 / 60 天 / 90 天
```

预测服务支持两种模式：

1. **Mock 预测模式**：用于项目联调和演示；
2. **外部模型接入模式**：队友只需提供标准预测结果 CSV，即可无缝接入真实模型结果。

---

### 6.5 红橙黄绿预警

预警等级基于 `risk_30d` 和 `risk_90d`：

| 预警等级 | 触发条件 | 建议巡检窗口 |
|---|---|---|
| 红色预警 | risk_30d ≥ 0.75 或 risk_90d ≥ 0.90 | 3～7 天内 |
| 橙色预警 | risk_30d ≥ 0.55 或 risk_90d ≥ 0.75 | 7～14 天内 |
| 黄色预警 | risk_30d ≥ 0.35 或 risk_90d ≥ 0.55 | 14～30 天内 |
| 绿色关注 | 其他 | 常规周期 |

---

### 6.6 维修建议

维修建议不是最终根因诊断，而是巡检方向参考。

建议来源包括：

- 历史工单中的故障描述；
- 关键词匹配规则；
- 后续可接入维修知识库/RAG；
- 风险预测和预警等级。

输出内容包括：

- 故障现象；
- 可能原因；
- 检查方向；
- 备件建议；
- 后续维护建议。

---

### 6.7 AFC 诊断 Agent

Agent 不负责训练预测模型，也不直接读取 Excel。

Agent 的职责是：

1. 理解用户问题；
2. 识别设备编号；
3. 判断任务类型；
4. 选择并调用工具；
5. 整合工具返回结果；
6. 生成结构化诊断报告；
7. 输出科学边界说明；
8. 使用 LangSmith 记录诊断过程。

Agent 的合理定位是：

> 一个面向 AFC 运维场景的诊断型 Agent，而不是通用聊天机器人。

---

## 7. 为什么不设计很多子 Agent？

当前项目不适合拆成大量子 Agent。

原因：

1. AFC 运维诊断链路较清晰，不需要多个 Agent 自由协商；
2. 多 Agent 会增加状态传递、Prompt 管理和调试复杂度；
3. 面试时容易被追问“为什么必须拆这么多 Agent”；
4. 对当前 MVP 来说，一个中心诊断 Agent 加多个工具更稳定、更容易落地；
5. LangGraph 的价值在于流程可控，不在于 Agent 数量多。

因此，本项目采用：

```text
一个 AFCDiagnosisAgent
+ 一组 LangChain Tools
+ 一个 LangGraph 诊断流程
+ LangSmith Trace / Evaluation
```

---

## 8. Agent 的核心价值

不用 Agent 时，用户需要分别查看：

1. 数据概览页面；
2. 设备列表页面；
3. 单设备历史页面；
4. 风险预测页面；
5. 维修建议页面；
6. 自己整理结论。

使用 Agent 后，用户只需要自然语言提问：

> “帮我分析设备 100023，看看未来一个月风险高不高，应该先检查什么。”

Agent 自动完成：

- 设备识别；
- 任务理解；
- 工具选择；
- 风险预测；
- 预警判断；
- 历史摘要；
- 维修建议；
- 诊断报告生成。

因此，Agent 的价值是：

> 把多个页面、多个接口、多个工具的结果，自动组织成一个面向运维决策的诊断结论。

---

## 9. 科学边界声明

系统必须明确说明：

1. `current_faildate` 表示工单记录时间，不一定等于物理故障发生时刻；
2. 风险预测表示未来时间窗口内再次产生故障工单的风险，不等于设备一定会在某一天发生故障；
3. 维修建议是巡检方向参考，不是最终根因诊断结论；
4. 最终维修判断仍需结合现场检测、设备日志和人工经验。

---

## 10. 面试讲解口径

可以这样介绍项目：

> 这个项目是一个面向地铁 AFC 设备的智能运维系统。它基于历史故障工单数据，预测设备在未来 7、14、21、30、60、90 天内再次产生故障工单的风险，并根据 30 天和 90 天风险生成红橙黄绿预警。  
>  
> 我的重点不是训练预测模型，而是做系统工程和 Agent 编排。我把数据概览、设备历史、风险预测、预警等级、维修建议封装成工具，再用 LangGraph 设计一个 AFC 诊断 Agent。用户可以用自然语言询问某台设备风险高不高、为什么触发预警、应该先检查什么，Agent 会调用对应工具，整合证据，生成结构化诊断报告。  
>  
> 同时我计划接入 LangSmith 追踪每次 Agent 调用了哪些工具、拿到了什么结果、最终报告是否忠实使用工具输出，从而保证系统不是黑盒聊天，而是可追踪、可解释、可评估的运维诊断流程。

## 11. 第一阶段开发目标

第一阶段不要追求复杂模型和复杂 RAG，先完成：

1. FastAPI 后端基础接口；
2. Streamlit 前端页面；
3. 工单上传与读取；
4. 数据概览；
5. 设备列表与单设备历史；
6. Mock 风险预测；
7. 红橙黄绿预警；
8. 维修建议；
9. 单设备综合分析；
10. AFCDiagnosisAgent 的 LangGraph 工作流骨架；
11. LangSmith Trace 接入。

---

## 12. 后续升级方向

| 模块 | 第一阶段 | 后续升级 |
|---|---|---|
| 预测模型 | Mock/外部 CSV | 队友真实 ML 模型 |
| Agent | 单 Agent + 工具调用 | LangGraph 完整状态流 |
| RAG | 预留接口 | 维修手册/历史案例向量检索 |
| 数据存储 | 文件系统 | PostgreSQL / MySQL |
| 向量库 | 暂不强制 | Chroma / FAISS / pgvector |
| 观测评估 | LangSmith Trace | Dataset + Evaluation |
| 前端 | Streamlit | 可升级 React/Vue |
| 部署 | 本地运行 | Docker + 云部署 |

---

## 13. 最终项目主目标

最终目标不是做一个“看起来会聊天”的系统，而是做一个：

> 能基于工单数据、预测结果和维修知识，帮助地铁 AFC 运维人员判断设备复发风险、解释预警原因、制定巡检方向的可解释 Agent 系统。
