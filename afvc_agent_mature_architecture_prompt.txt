============================================================
一、本轮最重要的架构原则
============================================================

不要新增新的 LangGraph 节点。

请保持当前八节点主链路：

prepare_context
→ understand_query
→ plan_tools
→ execute_tools
→ merge_evidence
→ evaluate_evidence
→ generate_answer
→ update_memory

不要新增：

route_query
guard_query
chat_answer
ask_clarification

这类新节点。

成熟 Agent 产品不是遇到一个边界情况就加一个节点，而是保持主链路稳定，把新能力内聚到现有节点的 schema、策略、prompt、校验逻辑和回答模式里。

所以本轮要做的是：

- 闲聊兜底放进 understand_query 的路由识别里
- 是否需要工具放进 plan_tools 的规划逻辑里
- 缺少设备编号的拦截放进 plan_tools / execute_tools
- 工具错误整理放进 merge_evidence
- 工具错误是否影响回答放进 evaluate_evidence
- 最终闲聊、能力介绍、缺参数追问、业务诊断回答，都放进 generate_answer

============================================================
二、当前问题背景
============================================================

现在系统存在一个典型问题：

用户只输入：

你好

系统却可能走成：

understand_query: task_type = unknown
plan_tools: 默认调用 get_integrated_analysis_tool
execute_tools: assetnum = None
工具报错：assetnum Input should be a valid string
evaluate_evidence: 仍然认为 evidence sufficient

这是不成熟的 Agent 行为。

正确行为应该是：

你好
→ understand_query: route = direct_chat
→ plan_tools: answer_mode = direct_chat, tool_calls = []
→ execute_tools: 无工具，跳过
→ merge_evidence: 空证据包
→ evaluate_evidence: direct_chat 不需要业务证据，可以回答
→ generate_answer: 返回自然问候 + 系统能力介绍
→ update_memory: 保留上下文，不污染业务记忆

============================================================
三、不要把 understand_query 做成鸡肋的精细意图分类器
============================================================

当前项目里 understand_query 可能过度依赖很多细碎的 task_type，例如：

risk_query
history_query
advice_query
risk_explanation
risk_and_advice_query
manual_query
followup_rewrite
unknown

这会导致系统像传统客服机器人一样做“意图分类游戏”。

本轮请把 understand_query 改造成更成熟的：粗粒度语义路由 + 参数提取器。

它不需要把每句话都强行分成十几个细意图。它应该判断：

1. 用户是不是闲聊 / 问候
2. 用户是不是问系统能力 / 怎么使用
3. 用户是不是问全局数据问题
4. 用户是不是问单设备业务问题
5. 用户是不是缺少设备编号
6. 用户是不是超出系统能力
7. 用户有没有设备编号
8. 用户有没有时间窗口
9. 用户是否需要工具
10. 是否可以直接回答

============================================================
四、建议新的 QueryUnderstanding 结构
============================================================

请尽量在不破坏现有 API 的基础上，扩展或重构 QueryUnderstanding。

建议增加或调整这些字段：

{
  "route": "direct_chat | capability_query | business_global | business_device | needs_clarification | unsupported",
  "business_goal": "data_overview | high_risk_ranking | device_risk | device_history | device_advice | full_diagnosis | manual_search | null",
  "assetnum": "1000029970 或 null",
  "time_window": "7d | 14d | 21d | 30d | 60d | 90d | null",
  "needs_asset": true,
  "needs_tools": true,
  "needs_rag": false,
  "context_used": false,
  "information_need": "用户想查询设备未来30天复发风险",
  "user_question_rewrite": "查询设备1000029970未来30天复发风险",
  "confidence": 0.9
}

如果为了兼容前端或现有代码仍需保留 task_type，可以保留，但不要再让它承担核心路由职责。

可以做映射：

route = direct_chat          → task_type = capability_query 或 direct_chat
route = capability_query     → task_type = capability_query
route = business_global      → task_type = data_overview / high_risk_ranking
route = business_device      → task_type = risk_query / history_query / advice_query / full_diagnosis
route = needs_clarification  → task_type = unknown
route = unsupported          → task_type = unknown

但内部主逻辑应优先使用 route 和 business_goal。

============================================================
五、route 的语义定义
============================================================

1. direct_chat

适用于：

你好
hello
hi
在吗
早上好
谢谢
辛苦了
你真厉害
随便聊聊

这类不涉及 AFC 业务数据的问题。

处理方式：不调用任何业务工具。最终由 generate_answer 生成自然回复，可顺带简短介绍系统能做什么。

2. capability_query

适用于：

你是谁
你能做什么
这个系统怎么用
你可以分析什么
支持哪些功能
怎么查询设备风险

处理方式：不调用业务工具。最终由 generate_answer 返回系统能力介绍和示例问题。

3. business_global

适用于全局业务问题：

数据概览
当前有多少工单
有哪些线路
有哪些品牌
故障类型分布
高风险设备有哪些
风险最高的设备 Top 10
哪些设备优先巡检

对应 business_goal：

data_overview
high_risk_ranking

处理方式：需要工具。data_overview 调用 get_data_summary_tool。high_risk_ranking 调用 get_high_risk_devices_tool。

4. business_device

适用于明确单设备问题：

设备1000029970风险怎么样
分析设备1000029970
1000029970最近有什么故障
给设备1000029970维修建议
设备1000029970为什么风险高

对应 business_goal：

device_risk
device_history
device_advice
full_diagnosis
manual_search

处理方式：需要工具。必须有 assetnum。如果当前输入没有 assetnum，但上下文有 last_assetnum，可以继承。如果没有 assetnum，也没有可继承上下文，应改为 needs_clarification。

5. needs_clarification

适用于用户明显想做业务分析，但缺少关键参数。例如：

帮我分析一下这个设备
看看它风险高不高
给我维修建议
这个怎么处理
最近怎么样

如果上下文有 last_assetnum，可以继承设备编号后转成 business_device。

如果上下文没有 last_assetnum，则：不调用工具，answer_mode = ask_for_assetnum，最终回复请用户提供设备编号，例如 1000029970。

6. unsupported

适用于超出系统能力的问题：

帮我写论文
今天北京天气怎么样
帮我算数学题
推荐电影

处理方式：不调用业务工具。最终回复说明当前系统主要面向地铁 AFC 故障复发风险预测、历史工单分析、高风险排行和维修建议，并引导用户换成相关问题。

如果是普通闲聊，不要过度拒绝，可以用 direct_chat。

============================================================
六、plan_tools 的成熟设计
============================================================

plan_tools 不应该只做 task_type 到 tool 的硬编码映射。

它应该根据 route + business_goal + assetnum 决定：

1. 是否需要工具
2. 调用哪些工具
3. 是否需要补充设备编号
4. 最终回答模式是什么

请扩展 ToolPlan，建议增加字段：

{
  "tool_calls": [],
  "use_existing_evidence": false,
  "reason": "用户为普通问候，不需要调用业务工具",
  "answer_mode": "direct_chat | capability_intro | ask_for_assetnum | evidence_based | unsupported",
  "answer_policy": {
    "must_not_predict_exact_failure_date": true,
    "must_answer_with_risk_window": false
  }
}

answer_mode 是本轮重点字段。

answer_mode 定义：

1. direct_chat：普通闲聊，不调用工具。
2. capability_intro：系统能力介绍，不调用工具。
3. ask_for_assetnum：用户想做设备业务分析，但缺少设备编号，不调用工具，直接追问设备编号。
4. evidence_based：业务问题，需要调用工具，并基于 EvidencePacket 回答。
5. unsupported：超出系统能力，不调用工具，礼貌说明边界。

============================================================
七、plan_tools 的具体规则
============================================================

1. direct_chat

route = direct_chat
→ tool_calls = []
→ answer_mode = direct_chat

2. capability_query

route = capability_query
→ tool_calls = []
→ answer_mode = capability_intro

3. needs_clarification

route = needs_clarification
→ tool_calls = []
→ answer_mode = ask_for_assetnum

4. unsupported

route = unsupported
→ tool_calls = []
→ answer_mode = unsupported

5. business_global + data_overview

route = business_global
business_goal = data_overview
→ call get_data_summary_tool
→ answer_mode = evidence_based

6. business_global + high_risk_ranking

route = business_global
business_goal = high_risk_ranking
→ call get_high_risk_devices_tool
→ answer_mode = evidence_based

7. business_device + device_risk

route = business_device
business_goal = device_risk
assetnum exists
→ call predict_device_risk_tool
→ answer_mode = evidence_based

8. business_device + device_history

route = business_device
business_goal = device_history
assetnum exists
→ call get_device_history_tool
→ answer_mode = evidence_based

9. business_device + device_advice

route = business_device
business_goal = device_advice
assetnum exists
→ call get_maintenance_advice_tool
→ answer_mode = evidence_based

10. business_device + full_diagnosis

route = business_device
business_goal = full_diagnosis
assetnum exists
→ call get_integrated_analysis_tool
→ answer_mode = evidence_based

11. business_device 但 assetnum 缺失

route = business_device
assetnum missing
→ tool_calls = []
→ answer_mode = ask_for_assetnum

12. unknown / fallback

不要再默认调用 get_integrated_analysis_tool。

如果确实无法判断：tool_calls = []，answer_mode = direct_chat 或 unsupported，具体取决于用户输入是否像闲聊。

============================================================
八、execute_tools 的安全要求
============================================================

execute_tools 是确定性代码节点，不用 LLM。

它只执行 tool_calls。

如果 tool_calls = []，应该正常跳过，不能报错。

请加入工具参数保护。

这些工具必须有有效 assetnum：

get_integrated_analysis_tool
predict_device_risk_tool
get_device_history_tool
get_maintenance_advice_tool

如果缺少 assetnum：不要调用工具。返回结构化 tool error。错误信息应清楚说明：缺少设备编号，无法执行该工具。

不要再出现 Pydantic 的底层报错直接暴露给前端，例如：

assetnum Input should be a valid string

应该转换成用户可理解、调试可读的错误：

{
  "status": "error",
  "error_type": "missing_required_argument",
  "tool": "get_integrated_analysis_tool",
  "message": "调用综合分析工具需要设备编号 assetnum，但当前问题没有提供设备编号。"
}

============================================================
九、merge_evidence 的改造要求
============================================================

请让 EvidencePacket 能承载工具错误。

建议在 EvidencePacket schema 里增加：

{
  "tool_errors": [
    {
      "tool": "get_integrated_analysis_tool",
      "error_type": "missing_required_argument",
      "message": "缺少设备编号"
    }
  ]
}

如果不想改 schema，也至少要把错误放进 missing_evidence 或一个清晰字段中，供 evaluate_evidence 使用。

要求：

1. 工具成功结果进入对应证据字段
2. 工具失败结果进入 tool_errors
3. 不要把失败工具当成有效证据
4. 不要因为 tool_calls = [] 就认为异常
5. 对 direct_chat / capability_intro / ask_for_assetnum，空证据包是正常的

============================================================
十、evaluate_evidence 的改造要求
============================================================

evaluate_evidence 不能再无脑判断 Evidence is sufficient to answer the question。

它必须结合 answer_mode 判断。

如果 answer_mode 是 direct_chat / capability_intro / ask_for_assetnum / unsupported：不需要业务证据，answerable = true，need_more_tools = false。

如果 answer_mode 是 evidence_based：必须检查证据是否足够。

例如：

- 用户问风险，但没有 risk_prediction → 不足
- 用户问历史，但没有 history_summary → 不足
- 用户问维修建议，但没有 maintenance_advice → 不足
- 用户问综合诊断，但缺少风险 / 历史 / 建议中的关键证据 → 不足
- 工具报错且没有有效证据 → 不足

如果工具报错是缺少设备编号，通常不要继续补工具，应生成 ask_for_assetnum 类型回答。

请保留最大工具补充循环次数，避免无限循环。

============================================================
十一、generate_answer 的改造要求
============================================================

generate_answer 应根据 answer_mode 输出。

1. direct_chat

用户问“你好”时，应该回复类似：

你好，我是地铁 AFC 故障复发风险预测与智能维修建议助手。你可以让我查看某台设备的历史故障、未来风险、高风险设备排行、维修建议或综合诊断。

不要调用工具，不要输出错误。

2. capability_intro

用户问“你能做什么？”时，应该回复系统能力说明：

我可以帮助你完成：
1. 查看 AFC 设备历史故障
2. 预测未来 7 / 14 / 21 / 30 / 60 / 90 天复发风险
3. 查看高风险设备 Top N
4. 生成单设备综合诊断
5. 根据故障历史给出维修巡检建议
6. 查看当前工单数据概览

示例：
“分析设备 1000029970”
“查看高风险设备 Top 10”
“设备 1000029970 最近有什么故障？”

3. ask_for_assetnum

用户想分析设备但上下文没有设备编号时，回复：

可以，请先提供设备编号，例如 1000029970。拿到设备编号后，我可以查询它的历史故障、未来复发风险、预警等级和维修建议。

4. unsupported

用户问系统外问题时，回复：

这个问题超出了当前 AFC 故障诊断系统的范围。我主要支持设备历史故障、复发风险、高风险排行、维修建议和数据概览分析。你可以提供设备编号让我继续分析。

5. evidence_based

业务问题必须基于 EvidencePacket 生成回答。

禁止编造：设备历史、风险数值、预警等级、维修建议、维修手册内容、具体故障日期。

必须保留科学边界：

本系统不能预测设备具体哪一天发生故障，只能给出未来时间窗口内的复发风险；维修建议是巡检方向参考，不等于最终根因判定。

============================================================
十二、update_memory 的要求
============================================================

update_memory 要避免闲聊污染业务上下文。

规则：

- direct_chat：不清空 last_assetnum，不覆盖 last_task_type 为 unknown，保留原业务上下文。
- capability_intro：不清空 last_assetnum，不覆盖 last_task_type 为 unknown。
- ask_for_assetnum：可以记录 last_task_type / pending_business_goal，不要设置 last_assetnum。
- business_global：不要错误继承上一轮设备编号，不要把 high_risk_ranking 绑定到某个设备。
- business_device：更新 last_assetnum，更新 last_task_type / business_goal，更新证据摘要。

文档里必须讲清楚：

1. 当前 Agent 架构

明确当前是八节点：

prepare_context
→ understand_query
→ plan_tools
→ execute_tools
→ merge_evidence
→ evaluate_evidence
→ generate_answer
→ update_memory

2. 不新增节点的成熟设计原则

当前架构通过扩展 QueryUnderstanding、ToolPlan、EvidencePacket 和 answer_mode 支持闲聊、能力介绍、缺参数追问和业务诊断，不通过不断新增节点解决边界问题。

3. 哪些节点用 LLM

understand_query：LLM 参与粗粒度语义路由和参数提取
plan_tools：LLM 参与工具规划，但工具调用经过白名单和参数校验
evaluate_evidence：LLM 参与证据充分性判断
generate_answer：LLM 参与最终回答生成

4. 哪些节点不用 LLM

prepare_context：确定性整理上下文
execute_tools：确定性执行工具
merge_evidence：确定性整理证据
update_memory：确定性更新多轮记忆

5. answer_mode 设计

文档中要解释：

direct_chat
capability_intro
ask_for_assetnum
evidence_based
unsupported

并说明它们如何避免“你好也调用诊断工具”的问题。

6. 工具和证据边界

业务问题必须走工具和 EvidencePacket。LLM 不允许绕过工具直接编造设备风险、历史工单、维修建议或手册内容。

7. 科学边界

系统不能预测设备具体哪一天发生故障，只能给出未来 7 / 14 / 21 / 30 / 60 / 90 天窗口内的复发风险。维修建议是巡检方向参考，不等于最终根因判定。

8. 当前未做内容

本轮没有接入真实机器学习预测模型。当前预测仍然是 baseline / mock / CSV adapter。本轮没有升级向量数据库 RAG。当前维修手册检索仍然是轻量关键词检索。本轮没有做 Docker / 云部署。
